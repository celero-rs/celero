Celero supports two project templates: **app** (modular) and **minimal** (single file).

## Minimal Template

Generated by `celero new my-api --template minimal`:

```
my-api/
├── Cargo.toml      # Dependencies
├── .gitignore
└── src/
    └── main.rs     # Everything in one file
```

Best for: prototypes, small services, FaaS functions.

## App Template

Generated by `celero new my-api --template app`:

```
my-api/
├── Cargo.toml
├── .env                    # Environment variables
├── .gitignore
└── src/
    ├── main.rs             # Entry point
    ├── lib.rs              # Module declarations
    ├── config/
    │   ├── mod.rs
    │   ├── settings.rs     # Configuration from env vars
    │   └── routes.rs       # Central route collection
    └── apps/
        ├── mod.rs          # App module registry
        └── core/
            ├── mod.rs
            ├── handlers/
            │   ├── mod.rs  # routes() function
            │   └── health.rs
            ├── schemas/    # Request/response DTOs
            ├── entities/   # SeaORM models (with ORM)
            ├── services/   # Business logic (with ORM)
            └── migrations/ # Database migrations (with ORM)
```

Best for: production APIs, microservices, multi-module applications.

## App Template — Key Files

### `main.rs`

The entry point connects the database, configures middleware, and starts the server:

```rust
use celero::prelude::*;

mod config;
mod apps;

#[tokio::main]
async fn main() {
    dotenvy::dotenv().ok();
    let settings = config::settings::Settings::from_env();

    let db = DbPool::connect(&settings.database_url).await.unwrap();

    let app = Celero::new()
        .title(&settings.app_name)
        .version("0.1.0")
        .database(db)
        .middleware(Cors::permissive())
        .middleware(RequestId)
        .routes(config::routes::collect_routes())
        .swagger("/docs")
        .build();

    app.serve(&format!("{}:{}", settings.host, settings.port))
        .await
        .unwrap();
}
```

### `config/settings.rs`

Loads configuration from environment variables with sensible defaults:

```rust
pub struct Settings {
    pub app_name: String,       // APP_NAME (default: project name)
    pub host: String,           // HOST (default: 0.0.0.0)
    pub port: u16,              // PORT (default: 3456)
    pub database_url: String,   // DATABASE_URL
    pub log_level: String,      // LOG_LEVEL (default: info)
}
```

### `config/routes.rs`

Central collection point for routes from all app modules:

```rust
pub fn collect_routes() -> Vec<RouteDefinition<DbPool>> {
    let mut all = Vec::new();
    all.extend(apps::core::handlers::routes());
    // [celero:app-routes]  <- marker for `celero add`
    all
}
```

### `.env`

```env
APP_NAME=my-api
HOST=0.0.0.0
PORT=3456
DATABASE_URL=sqlite://./app.db?mode=rwc
LOG_LEVEL=info
```

## Adding App Modules

Use `celero add` to create new app modules:

```bash
celero add users
celero add posts
```

Each command creates the module structure, updates `apps/mod.rs`, and wires routes into `config/routes.rs` automatically.

See [celero add](../cli/add.md) for details.

## Django-Inspired Architecture

The app-based structure is inspired by Django, adapted for Rust:

| Directory | Purpose | When Present |
|-----------|---------|--------------|
| `handlers/` | HTTP handlers with `#[api]` macro | Always |
| `schemas/` | Request/response DTOs with `#[derive(Schema)]` | Always |
| `entities/` | SeaORM entity models | With ORM |
| `services/` | Business logic and database queries | With ORM |
| `migrations/` | Database schema migrations | With ORM |

### Request Flow

```
Request -> Middleware -> Router -> Handler -> Service -> Entity/DB -> Response
```

- **Handler** — receives the request, validates input via Schema, delegates to Service
- **Service** — business logic, executes queries via Entity
- **Entity** — direct database table mapping (SeaORM)
- **Schema** — defines input/output shape (OpenAPI-aware)
